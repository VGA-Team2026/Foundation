# GitHub Issue ログ調査コマンド

GitHub Issueに貼られたログ（テキストや画像）を元に問題を調査・分析するコマンド。

## ロール
あなたはゲームプログラマです。
GitHub Issueに投稿されたログやスクリーンショットを分析し、問題の原因特定と対応方針の提示を行います。

## モデル使用方針
- **分析・判断**: Opusで行う（このコマンドを実行するメインエージェント）
- **コード調査**: Sonnetを使用したサブエージェント（Explore / general-purpose）を起動して行う
- サブエージェントには具体的な調査タスクを指示し、結果を受け取って分析する

## 引数
コマンド引数には `{Issue番号}` が渡される。
- 例: `/digissue 123`
- Issue番号がない場合は、オープン中のIssue一覧を表示して選択を促す。

## 処理フロー

### 1. Issue情報の取得
```bash
gh issue view {Issue番号} --json title,body,labels,assignees,milestone,comments
```
- Issueのタイトル、本文、ラベル、担当者、マイルストーン、**コメント**を全て取得する
- **重要: Issueのコメントは全て取得して読むこと**（追加のログや情報が含まれている可能性がある）

### 2. ログ・画像の抽出
Issue本文とコメントから以下を抽出する:

**重要: ログはリンクで貼られていることが多いため、必ずダウンロードすること。**
- GitHub Issueに添付されたファイルリンク（`.txt`, `.log`等）はWebFetchまたは`curl`でダウンロードする
- `https://github.com/user-attachments/assets/...` 形式のリンクはファイル添付
- ダウンロードコマンド例:
```bash
# GitHub添付ファイルのダウンロード
curl -L -o /tmp/issue_log.txt "{ログファイルURL}"
```

#### テキストログの抽出
- ダウンロードしたログファイルの内容を読み込む
- コードブロック内のログ（```で囲まれた部分）
- `[INFO]`, `[WARN]`, `[ERROR]`, `[EXCEPTION]` を含む行
- スタックトレース（`at ` で始まる行）
- Unity固有のログパターン（`UnityEngine.`, `NullReferenceException` 等）

#### 画像の抽出
- Issue内の画像URL（`![...](...)`形式）を取得
- GitHub Issueの画像添付（`https://github.com/user-attachments/assets/...`等）
- スクリーンショットがある場合はWebFetchで取得して分析

### 3. ログの分析

#### 3-1. エラー・例外の特定
テキストログから以下を検索:
- `NullReferenceException` - null参照
- `MissingReferenceException` - 削除済みオブジェクト参照
- `IndexOutOfRangeException` - 配列範囲外
- `[ERROR]`, `[EXCEPTION]` タグ
- `Exception`, `Error` を含む行
- スタックトレースからクラス名・メソッド名・行番号を抽出

#### 3-2. 関連コードの特定
エラーや問題に関連するコードを特定する:

```
Task(subagent_type="Explore", model="sonnet", prompt="
以下のエラー/ログから関連するソースコードを検索してください。

## エラー/ログ情報
{抽出したログ・エラー情報}

## 検索対象
1. スタックトレースに含まれるクラス名・メソッド名からソースファイルを特定
2. エラーメッセージに含まれるログタグ（[GameManager]等）からソースを特定
3. 関連する仕様書を spec/code/ から検索

## 報告内容
- 関連ソースファイルのパスと該当行
- エラーが発生しうる条件
- 関連する仕様書のパス
")
```

#### 3-3. 問題の原因分析
抽出した情報とコード調査結果を突き合わせて:
1. エラーの発生条件を特定
2. 再現手順を推定
3. 修正方針を検討

### 4. 画像分析（画像がある場合）
Issue内にスクリーンショットや動画キャプチャがある場合:
- 画像URLをWebFetchで取得し、視覚的に分析
- UIの状態、エラーダイアログ、ゲーム画面の異常を確認
- ログ情報と画像の状況を照合

### 5. レポートの出力
調査結果をユーザーに報告する。

#### 報告フォーマット
```markdown
# Issue #{Issue番号} ログ調査レポート

## Issue情報
- タイトル: {タイトル}
- URL: {URL}

## 抽出したログ・エラー
{主要なエラーやログの抜粋}

## 関連コード
| ファイル | 行 | 内容 |
|---------|-----|------|
| {パス} | {行番号} | {該当コードの説明} |

## 原因分析
{エラーの原因と発生条件の分析}

## 再現手順（推定）
1. {手順1}
2. {手順2}
...

## 修正方針
- {方針1}
- {方針2}

## 次のアクション
- [ ] {対応項目1}
- [ ] {対応項目2}
```

### 6. 対応の提案
分析結果に基づいて、ユーザーに次のアクションを提案する:
- コード修正が必要な場合: `/issue {番号}` で修正対応を提案
- 追加調査が必要な場合: 調査すべきポイントを提示
- ログが不足している場合: 追加情報のリクエストをIssueにコメント

## ログパターンリファレンス

### Unityログ形式
```
[HH:mm:ss.fff][ログレベル] [フレーム番号] [メッセージ]
```

### よくあるエラーパターン
| パターン | 意味 |
|---------|------|
| `NullReferenceException` | null参照アクセス |
| `MissingReferenceException` | 破棄済みオブジェクト参照 |
| `IndexOutOfRangeException` | 配列/リスト範囲外アクセス |
| `InvalidOperationException` | 不正な操作 |
| `KeyNotFoundException` | Dictionary未登録キーアクセス |
| `OperationCanceledException` | UniTask/CancellationTokenキャンセル |

### ゲーム固有のログタグ
| タグ | コンポーネント |
|-----|--------------|
| `[GameManager]` | ゲーム全体の制御 |
| `[Player]` | プレイヤー関連 |
| `[StagePresenter]` | ステージ制御 |
| `[StageBehaviour]` | ステージ動作 |
| `[HitDetector]` | 当たり判定 |
| `[Enemy]` | 敵関連 |
| `[BGM]` | BGM制御 |
| `[CaveDirector]` | 洞窟演出 |
| `[GoalDirector]` | ゴール演出 |

## 注意事項
- Issueに複数のログが貼られている場合は、全てのログを分析すること
- コメントに追加情報がある場合は必ず考慮すること
- 画像が貼られている場合は必ず確認すること
- 推測に基づく分析は「推定」と明記すること
- 修正コードは書かない（調査・分析のみ）。修正は `/issue` コマンドで行う
